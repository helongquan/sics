https://devdocs.magento.com/system-requirements.html

7.0.32 / PHP 7.0.32

安装magento2之前我们要做些准备，这个准备是笔者在多次失败后的失败的经验总结出来的。如果不做些准备工作，你可能碰到如下的问题。

  1. 可能安装不成功。
  2. 可能安装成功后，后台icon丢失，以及后台的链接点击不了。
  3. 可能前台样式错乱。

我们要做哪些准备工作呢？

第一，我们来到xampp那里，在php.ini文件中开启xampp的一些magento必备扩展，比如xsl,intl,soap等，如果这些必备的扩展没有开启就会没法进行下一步的安装，当然php版本必须符合。然后就是设置一些值，比如

memory_limit=1280M          //默认是128M
max_execution_time=600   //默认是60
max_input_time=600         //默认是60
post_max_size=128M        //默认是2M
upload_max_filesize=64M   //默认是2M

修改下my.ini的配置，比如：

max_allowed_packet = 256M    //默认是16M
myisam_sort_buffer_size = 256M     //默认是8M
等等...

httpd.conf里面也可以根据的一些需求进行修改，这里就不多说，具体的大家谷歌一下。

第二，碰到很多问题的可能是安装完magento2后，来到后台里面icon显示不出来，也就是字体图标没了，另外一个问题就是点击后台的那些菜单没反应，这个是最要命的。为了解决这个问题，笔者找了很多方案，这里必须提醒，这种问题一定要去google，而不是百度，百度里面基本很难找到关于magento的解决方案，果然笔者在google找到了很多方案。

确保这个问题搞定后，我们在来到/app/etc/di.xml，打开文件，搜索找到，

Magento\Framework\App\View\Asset\MaterializationStrategy\Symlink

替换为，

Magento\Framework\App\View\Asset\MaterializationStrategy\Copy

记得是替换为，因为笔者搜索的时候，确实搜索到了这个，可是就在隔壁就有了一个，

Magento\Framework\App\View\Asset\MaterializationStrategy\Copy

当时就是觉得直接把，

Magento\Framework\App\View\Asset\MaterializationStrategy\Symlink

这个注释掉即可，后来发现这两个的标签的属性是不一样的。笔者粘贴一下：

<argument name="strategiesList" xsi:type="array">
    <item name="view_preprocessed" xsi:type="object">Magento\Framework\App\View\Asset\MaterializationStrategy\Symlink</item>
    <item name="default" xsi:type="object">Magento\Framework\App\View\Asset\MaterializationStrategy\Copy</item>
</argument>

所以，正确的做法是替换掉Magento\Framework\App\View\Asset\MaterializationStrategy\Symlink

笔者在折腾magento2后台的时候，遇到这样一个奇怪的现象，当我把某个分类勾选更多的产品划分到这个分类或者把该分类去掉一些产品的时候，又或者我要修改某个分类的名字的时候，报了一个错误：

URL key for specified store already exists

后来笔者新建一个分类也是报这个错误，我觉得很奇怪，根据字面意思那么就是这个url已经存在，可是后面自己把名字更换了还是报错，后来查了下资料发现这个是一个重写的问题，根据提示笔者来到magento的数据库，把重写规则删除了。

首先登录数据库，然后：

select * from url_rewrite;

看下这个里面有哪些东西，发现里面有很多重写的。由于笔者的magento的数据都是导入的demo数据，所以不用担心，基本可以全部删除了。为了谨慎起见，笔者先把这个表备份了，然后删除某个有问题的，比如笔者修改分类为pen的这个报错了，那么可以把这个有关的都删除了。运行命令：

DELETE FROM url_rewrite WHERE request_path LIKE '%pen%';

抽象一下就是：

DELETE FROM url_rewrite WHERE request_path LIKE '%category-name%';     //把分类名作为关键词删除即可

笔者后来经过尝试发现删除分类名貌似没有起到立竿见影的作用，于是就用了粗暴的方法：

DELETE FROM url_rewrite WHERE request_path LIKE '%a%';
DELETE FROM url_rewrite WHERE request_path LIKE '%b%';
DELETE FROM url_rewrite WHERE request_path LIKE '%c%';
...

这些，逐级删除，后来感觉删得差不多了就把缓存清除一下，后面就OK了。

提示：这个毕竟是直接操作数据库，以防万一，先备份，后操作。

有的人说造成这个问题的原因还有另外一种解释，就是说有相同的分类名在不同的根分类下，原文：

Same category names under different root categories

参考链接：https://github.com/magento/magento2/issues/8304

Magento 2，无法登录后台，循环登录问题解决方法

就是当magento2安装完毕后，前台可是可以访问的，可是登录后台的时候，无论输入多少次都始终在登录界面：

进入一个循环登录界面。解决方案如下：

cd到magento2/vendor/magento/framework/Session目录，编辑SessionManager.php文件，搜索函数：expireSessionCookie()，注释如下代码：

$metadata->setHttpOnly($this->sessionConfig->getCookieHttpOnly());

然后在chrome浏览器里面登录貌似不行，在火狐浏览器（firefox）里可以登录。原因其实就是笔者用一个magento2安装成功后，然后复制了这个网站作为子域名进行安装，而这个子域名的网站的源码和数据跟主域名的源码和数据一模一样，而笔者在安装完子域名之前就在chrome浏览器登录了主域名，自然就保存了主域名的cookie信息，而当安装完毕子域名这个magento2后，那么还是在chrome登录的时候，可能就存在cookie的冲突的问题，所以当笔者换成了一个重来没有登录过主域名的这个mageno网站的火狐浏览器登录子域名的magento后，自然就不会存在cookie冲突的问题，登录就不会失败。

当然，如果你很喜欢用chrome登录子域名的magento网站，那么最简单的方法就是删除chrome的所有cookie信息，或者是早chrome的无痕模式下登录子域名的magento的网站。笔者试过了，是成功的。

magento网站添加倒计时秒杀定时器，类似京东淘宝那种效果设计方法

今天来跟大家说下对magento网站设计一个倒计时功能，就是限时秒杀那种效果。

首先我们确定几个问题：

  1. 显示在什么地方，笔者这里想让这个东西显示在每个产品详情页。
  2. 如何更方便的维护以及设置。
  3. 正确的做法。

确定显示的地方

通常来说这种会选择在产品详情页，或者首页显眼的地方，其实在哪里选择显示问题不带，也就是一个调用的问题。

笔者今天让这个显示在每个产品的详情页的某个地方。

我们先找到产品详情页的模板文件：app/design/frontend/yourtheme/default/template/catalog/product/view.phtml

根据开发者工具，我们找一个合适位置，这里就找到：

<div class="product-info">
...
</div>

在这个后面，添加我们的内容。内容如下：

<div class="daojishi">
    <?php
    $attributes = $_product->getAttributes();
    if ($attributes['deadline']->getFrontend()->getValue($_product)) {
        // echo "<span class='dea'>".$attributes['deadline']->getFrontend()->getValue($_product)."</span>";
        echo "<span class='deadlines'>".strtotime($attributes['deadline']->getFrontend()->getValue($_product))."</span>";

        if ($this->getLayout()->createBlock('cms/block')->setBlockId('daojishi')->toHtml()) {
            echo $this->getLayout()->createBlock('cms/block')->setBlockId('daojishi')->toHtml();
        }
    }
    ?>
</div>

里面的代码有两个判断语句，大概意思就是，如果这个产品设置了截止日期就输出时间戳，

strtotime($attributes['deadline']->getFrontend()->getValue($_product))

如果后台静态cms块有内容就输出，因为静态块的输出是依赖截止日期是否设置的，所以这样的判断顺序是合理的。

好的，这里我们确定了位置，我们先不管里面的代码是怎么回事，后面会讲。

代码我们添加完了，如何更方便的维护

这里我们采用magento后台的静态CMS块，然后在模板文件那里调用这个静态块。

我们来到magento后台的CMS的static block那里新建一个，然后命名为：deadline

里面添加一控制倒计时显示的js代码和html代码，代码如下：

<div class="jd_secKill_left">

<script>
var sheding_time = jQuery(".deadlines").text();
console.log(sheding_time);
// sheding_time = parseInt(sheding_time) + 43200;
// console.log(sheding_time);

var timestamp = Date.parse(new Date());
console.log(timestamp/1000);

time = sheding_time - (timestamp/1000);

var intDiff = parseInt(time);//time是倒计时总秒数量
function timer(intDiff){
    window.setInterval(function(){
    var day=0,
        hour=0,
        minute=0,
        second=0;//时间默认值       
    if(intDiff > 0){
        day = Math.floor(intDiff / (60 * 60 * 24));
        hour = Math.floor(intDiff / (60 * 60)) - (day * 24);
        minute = Math.floor(intDiff / 60) - (day * 24 * 60) - (hour * 60);
        second = Math.floor(intDiff) - (day * 24 * 60 * 60) - (hour * 60 * 60) - (minute * 60);

    if (minute <= 9) minute = '0' + minute;
    if (second <= 9) second = '0' + second;

    jQuery('#hour_show').html('<s id="h"></s>'+hour+':');
    jQuery('#minute_show').html('<s></s>'+minute+':');
    jQuery('#second_show').html('<s></s>'+second);

    }, 1000);

    timer(intDiff);

这个就是在block里面的代码，这里讲解下：

首先采用jquery获取dom元素的内容，

var sheding_time = jQuery(".deadlines").text();

我们打印到前台的时候，你会发现显示的格式是年月日，magento后台就只能选择年月日，不能选择时间。

我们已经获取到了产品后台设置的自定义属性deadline的值，

这里需要说的是，为了能够对某个产品自定义设置截止时间，必须现在magento后台添加一个自定义属性，添加完毕后，我们就可以在新建一个产品的时候，就可以设置截止日期了，当截止日期设置后，这个数据就会被保存进数据库中。

我们就可以通过：

<?php

这个就是在前面说的那个模板文件中的代码，获取后台自定义属性的值，获取完了之后再获取当前的时间戳，

var timestamp = Date.parse(new Date());

这个是以毫秒作为单位的，如果了解更多，可以去站长工具的时间戳切换工具中了解更多。

明确了这个之后呢，由于magento后台不能设置时间，只能设置年月日，所以这里只能想办法把毫秒变成秒为单位的。

timestamp/1000

然后，把这两个数进行相减，然后获得时间差，这个时间差就是这个限时的秒数总数了。

time = sheding_time - (timestamp/1000);

剩下的就是计算了，

timer(intDiff)

当然，我们也可以设置一个频率小点的，比如每个礼拜，每天执行一个特定的任务。比如每周备份，每周推送电邮消息。

具体的设置方法需要大家去了解下cron的表达式，也有cron生成器。这里科普一下cron表达是的用法以及意思：

|-|-|-|
|字段|允许值|允许的特殊字符|
|秒（Seconds）|0~59的整数|, - * / 四个字符|
|分（Minutes）|0~59的整数|, - * / 四个字符|
|小时（Hours）|0~23的整数|, - * / 四个字符|
|日期（DayofMonth）|1~31的整数（但是你需要考虑你月的天数）|,- * ? / L W C 八个字符|
|月份（Month）|1~12的整数或者 JAN-DEC|, - * / 四个字符|
|星期（DayofWeek）|1~7的整数或者 SUN-SAT （1=SUN）|, - * ? / L C # 八个字符|
|年(可选，留空)（Year）|1970~2099|, - * / 四个字符|

（1）*：表示匹配该域的任意值。假如在Minutes域使用*, 即表示每分钟都会触发事件。

（2）?：只能用在DayofMonth和DayofWeek两个域。它也匹配域的任意值，但实际不会。因为DayofMonth和DayofWeek会相互影响。例如想在每月的20日触发调度，不管20日到底是星期几，则只能使用如下写法： 13 13 15 20 * ?, 其中最后一位只能用？，而不能使用*，如果使用*表示不管星期几都会触发，实际上并不是这样。

（3）-：表示范围。例如在Minutes域使用5-20，表示从5分到20分钟每分钟触发一次 

（4）/：表示起始时间开始触发，然后每隔固定时间触发一次。例如在Minutes域使用5/20,则意味着5分钟触发一次，而25，45等分别触发一次. 

（5）,：表示列出枚举值。例如：在Minutes域使用5,20，则意味着在5和20分每分钟触发一次。 

（6）L：表示最后，只能出现在DayofWeek和DayofMonth域。如果在DayofWeek域使用5L,意味着在最后的一个星期四触发。 

（7）W:表示有效工作日(周一到周五),只能出现在DayofMonth域，系统将在离指定日期的最近的有效工作日触发事件。例如：在 DayofMonth使用5W，如果5日是星期六，则将在最近的工作日：星期五，即4日触发。如果5日是星期天，则在6日(周一)触发；如果5日在星期一到星期五中的一天，则就在5日触发。另外一点，W的最近寻找不会跨过月份 。

（8）LW:这两个字符可以连用，表示在某个月最后一个工作日，即最后一个星期五。 

（9）#:用于确定每个月第几个星期几，只能出现在DayofMonth域。例如在4#2，表示某月的第二个星期三。

注：

（1）有些子表达式能包含一些范围或列表

例如：子表达式（天（星期））可以为 "MON-FRI"，"MON"，"WED，FRI"，"MON-WED,SAT"

"*"字符代表所有可能的值

因此，"*"在子表达式（月）里表示每个月的含义，"*"在子表达式（天（星期））表示星期的每一天

"/"字符用来指定数值的增量 

例如：在子表达式（分钟）里的"0/15"表示从第0分钟开始，每15分钟 

在子表达式（分钟）里的"3/20"表示从第3分钟开始，每20分钟（它和"3，23，43"）的含义一样

database_backup.sh代码：

#!/bin/bash
# Name:bakmysql.sh
# This is a ShellScript For Auto DB Backup and Delete old Backup
#
backupdir="/home/noctiuni/public_html/website_backup/db_bak"
password="yourpassword"
time=`date "+%Y-%m-%d-%H:%M:%S"`
# backup database file to local
rm -rf $backupdir/*
mysqldump -u databaseusername -p$password noctiuni_www | gzip > $backupdir/noctiuni_www_$time.sql.gz

source_backup.sh代码：

#!/bin/bash
# Name:bakmysql.sh
# This is a ShellScript For Auto DB Backup and Delete old Backup
#
backupdir="/home/noctiuni/public_html/website_backup/magento1"
magento1_dir="/home/noctiuni/public_html/magento1"
time=`date "+%Y-%m-%d-%H:%M:%S"`
# backup database file to local
rm -rf $backupdir/*
zip -r $backupdir/magento1_$time.zip $magento1_dir/*

chmod -R 777 database_backup.sh
chmod -R 777 source_backup.sh

然后，就是编写crontab了。在任意一个目录下运行命令：

crontab -e

添加：

    */5 * * * * php /home/noctiuni/public_html/magento1/cron.php
    */10 17-20 * * * /home/noctiuni/public_html/website_backup/database_backup.sh
    */13 17-20 * * * /home/noctiuni/public_html/website_backup/source_backup.sh

第一行就是说，每隔五分钟执行一次magento根目录下的cron.php文件。

第二行就是每天下午的17点到20点，每隔10分钟执行一次脚本database_backup.sh

第三行就是每天下午的17点到20点之间每隔13分钟执行一次脚本source_backup.sh

    */6 * * * * /home/noctiuni/public_html/website_backup/database_backup.sh
    */8 * * * * /home/noctiuni/public_html/website_backup/source_backup.sh
